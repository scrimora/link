# Release Checklist

Scrimora Link ships through GitHub Releases. Tagged releases are signed for the
Tauri updater and produce the `latest.json` manifest consumed by installed apps.

## One-time GitHub setup

1. Make the repository public.
2. Confirm the repository is `scrimora/link`. The updater endpoint is:
   `https://github.com/scrimora/link/releases/latest/download/latest.json`.
3. Add these repository secrets:
   - `TAURI_SIGNING_PRIVATE_KEY`
   - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
   - `TAURI_UPDATER_PUBLIC_KEY`
4. Keep the private updater key secret. Only the public key belongs in build
   configuration or release environments.

## Optional OS signing

Updater signing verifies Scrimora Link updates, but it does not replace OS trust.
For a lower-friction public launch, also configure:

- macOS Developer ID signing and notarization.
- Windows Authenticode signing.

The current workflow can publish downloadable artifacts without OS code-signing
credentials, but users may see operating system trust warnings.

## Cutting a Release

1. Update the version in all three files:
   - `package.json`
   - `src-tauri/Cargo.toml`
   - `src-tauri/tauri.conf.json`
2. Run local verification:
   ```sh
   bun install --frozen-lockfile
   cargo fmt --check
   cargo clippy --all-targets -- -D warnings
   cargo test --locked
   ```
3. Commit the version bump.
4. Create and push the matching tag:
   ```sh
   git tag v0.1.0
   git push origin v0.1.0
   ```
5. Wait for the `release` workflow to complete.
6. Confirm the GitHub Release contains:
   - Windows installer assets.
   - macOS installer assets for Apple Silicon and Intel.
   - `latest.json`.
   - Signature artifacts generated by Tauri.

## Auto-update Notes

The app compiles the updater public key from
`SCRIMORA_LINK_UPDATER_PUBLIC_KEY` and derives its update endpoint from
`SCRIMORA_LINK_GITHUB_REPOSITORY` in CI. Release builds set both values in the
workflow, so installed apps check GitHub Releases for the newest stable release.
